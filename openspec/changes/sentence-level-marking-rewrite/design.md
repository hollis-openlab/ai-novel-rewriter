## Context

当前系统在 Analyze/Mark/Rewrite 链路里主要依赖 `paragraph_range`。这在结构化小说文本里可用，但在高频换行、对话密集、导入格式不稳定的章节中会放大命中误差和 diff 噪声。用户要求把“句号后换行”作为可选结果样式，同时明确不应破坏原始文本。

现状约束：
- 生产链路已经围绕 `paragraph_range + anchor hash` 运行，不能破坏兼容
- 运行状态逻辑近期已修复为 `resume` 补跑未完成章节，需要保持一致
- 前端对 diff 视图已具备 raw/canonical 对齐切换能力，可复用

## Goals / Non-Goals

**Goals:**
- 在不破坏现有段落链路的前提下，补齐句子级元数据（句子范围与字符偏移）
- 改写源文本切片优先走偏移，提升命中稳定性和可解释性
- 将“句末换行”下沉为展示/导出层可选模式，不污染核心存储
- 保持导入、分章、分析、改写、导出全链路向后兼容

**Non-Goals:**
- 不重写 Analyze 模型输出协议（仍接受 paragraph_range）
- 不在本次变更中替换所有锚点字段（paragraph hash 继续保留）
- 不一次性改造全部前端编辑交互，仅先补后端能力与关键状态语义

## Decisions

### 1) 双轨文本：Raw 持久化真值，Canonical 计算视图
决策：保留现有 Raw 文本存储不变，仅在计算环节引入 canonical 归一化和句子切分，不回写 Raw。

备选方案：
- 方案 A：导入后直接清洗覆盖原文（拒绝）
- 方案 B：双轨并行（采用）

理由：A 会破坏可追溯性和人工审校一致性；B 可最大化兼容并且便于回滚。

### 2) 标记阶段追加句子级元数据，不替换段落字段
决策：`RewriteSegment` 增加可选 `sentence_range` 与 `char_offset_range`，保留 `paragraph_range` 强兼容。

备选方案：
- 方案 A：直接把主键字段改为 sentence_range（拒绝）
- 方案 B：增量字段并双轨执行（采用）

理由：A 破坏性过大，测试与历史 artifact 成本高；B 可平滑迁移。

### 3) Rewrite 切片优先偏移，缺失时回退段落
决策： `_segment_source_text` 先尝试 `char_offset_range`，缺失/非法时回退现有段落切片。

备选方案：
- 方案 A：偏移唯一来源（拒绝）
- 方案 B：偏移优先 + 段落回退（采用）

理由：保证鲁棒性，避免旧数据无法改写。

### 4) 句末换行仅做展示/导出后处理
决策：新增可选 reflow 模式，对输出文本做句末换行，不改数据库与 stage artifact 原文。

备选方案：
- 方案 A：改写入库前重排版（拒绝）
- 方案 B：导出/展示时可选重排（采用）

理由：A 会让审核与追溯脱钩；B 保留数据层稳定。

### 5) 超长改写片段自动拆分执行并自动合并
决策：当单个 `RewriteSegment` 超过长度阈值或超过模型单次安全输出预算时，在 Rewrite 执行层按句末/段落边界自动拆分为多个子段串行改写；改写完成后自动拼接为单段 `rewritten_text` 返回，保持存储结构不变。

备选方案：
- 方案 A：继续单次请求（拒绝）
- 方案 B：在 mark 阶段直接拆成多个持久 segment（暂不采用）
- 方案 C：执行层临时拆分并回合并（采用）

理由：A 在 8k+ 中文章节上易触发截断；B 会改变审核与回退语义；C 不破坏现有 artifact 协议且可快速提升稳定性。

## Risks / Trade-offs

- [句子切分规则误判] → 先采用保守规则（。！？… + 引号闭合），并保留段落回退
- [偏移与原文不同步] → 通过回退路径兜底，记录 warning 并可视化
- [旧 artifact 不含新字段] → 全部字段设计为 optional，读取时自动兼容
- [导出样式与用户预期差异] → 提供开关并默认关闭，避免影响现有流程

## Migration Plan

1. 扩展模型：新增可选句子级字段（不破坏旧字段）
2. 扩展标记构建：写入句子范围/偏移
3. 扩展改写切片：偏移优先 + 回退段落
4. 扩展导出渲染：增加可选句末换行模式
5. 回归测试：旧 artifact、新 artifact、resume 补跑与状态流转

回滚策略：
- 如出现异常，仅需忽略新增字段并回退 `_segment_source_text` 偏移逻辑即可

## Open Questions

- 句子切分是否要支持更复杂书面语规则（如缩写、省略号内引号嵌套）？
- 句末换行是否应在 UI 中暴露为“章节预览模式”，还是先仅开放导出参数？
