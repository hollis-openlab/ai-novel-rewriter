## Frontend 三栏工作台低保真线框（逐屏 / 逐按钮 / 逐状态）

## 1. 目标与范围

本方案用于统一 Split / Analyze / Mark / Rewrite / Assemble 五个核心环节的前端交互。

设计目标：
1. 所有环节使用同一套「左中右三栏工作台」骨架，降低学习成本。
2. 用户始终知道“当前在哪一章”“当前阶段做什么”“下一步点哪里”。
3. 对比改写时提供 Git 风格红删绿增，支持人工审核决策闭环。
4. 所有长内容都在区域内滚动，不拉长整页。

不在本轮范围：
1. 具体 UI 视觉精修。
2. 组件开发与联调实现。
3. Docker 相关发布改造（明确顺延到最后一轮）。

## 2. 全局信息架构（统一骨架）

```
+--------------------------------------------------------------------------------------------------+
| 顶部条：小说标题 / 当前任务 / 当前 Stage / 全局筛选 / 快捷操作                                  |
+--------------------------------------------------------------------------------------------------+
| 左栏（章节导航）       | 中栏（主工作区）                                  | 右栏（智能侧栏）     |
|------------------------|--------------------------------------------------|----------------------|
| 章节搜索               | 二级工具条（视图切换、定位、缩放、过滤）        | Tab1 洞察            |
| 状态筛选               |--------------------------------------------------| Tab2 操作            |
| 章节列表（可滚动）     | 内容阅读/对比/编辑主区域（内部滚动）            | Tab3 日志            |
| 批量操作入口           | 底部浮动操作条（保存、确认、重试、接受/拒绝）    | 运行历史 / 快照      |
+--------------------------------------------------------------------------------------------------+
```

列宽建议：
1. 左栏：`280px`（最小 `240px`）。
2. 右栏：`360px`（最小 `320px`）。
3. 中栏：自适应，最小可用宽度 `720px`。

滚动规则：
1. 页面整体只在高度不足时滚动一层。
2. 左栏章节列表独立滚动。
3. 中栏正文/对比区域独立滚动。
4. 右栏面板独立滚动。

## 3. 共享组件定义（跨 Stage 复用）

### 3.1 左栏：章节导航卡片（Chapter Row）

显示字段：
1. `chapter_index + title`
2. `char_count`
3. 当前 Stage 状态徽章
4. 风险标识（warning / failed / stale）

状态徽章建议：
1. `pending`
2. `running`
3. `completed`
4. `failed`
5. `stale`
6. `skipped`
7. `accepted_edited`（Rewrite 子状态）

交互：
1. 单击：切换当前章节。
2. 上下键：切换上一章/下一章。
3. 支持筛选：`全部 / 待处理 / 失败 / 已完成 / 需人工确认`。

### 3.2 中栏：视图模式切换

统一按钮组：
1. `原文`
2. `改写稿`
3. `Diff（并排）`
4. `Diff（行内）`
5. `最终稿（Assemble 后）`

语义约束（避免“原文/改写稿看起来一样”）：
1. `原文`：仅显示章节原文快照（Split/Chapter 内容），不掺入改写结果。
2. `改写稿`：显示当前章节工作草稿（仅应用 `completed/accepted/accepted_edited` 的 segment）。
3. `Diff`：固定比较 `原文` vs `改写稿`，不得比较两个“改写稿派生视图”。
4. 当章节无可用改写（全 `failed/pending/rejected`）时，`改写稿` 必须展示空结果态与失败统计，不可静默回退成与原文一致的长文展示。

### 3.3 右栏：Tab 结构

1. `洞察`
- 章节元信息
- 规则命中/场景识别/风险告警
- 快照摘要（provider/model/rules hash）

2. `操作`
- 本阶段主按钮
- 单章操作（重试、跳过、回退）
- 批量操作（仅对筛选结果）
- Rewrite 仅放“动作入口”，不承载正文编辑输入框

3. `日志`
- 运行日志时间线
- prompt 调用记录
- 错误码和 stack 摘要

## 4. 逐屏低保真线框

### 4.1 Screen A：Split（切分）

```
左栏：章节（预览）列表
- [ ] 第1章 前言
- [ ] 第2章 第一章
- ...

中栏：
[规则选择下拉] [按选中规则预览] [确认切分]
------------------------------------------------
预览章节内容（全部章节，内部滚动）
  卡片：章号/标题/字数/摘要片段

右栏：
Tab 洞察：命中规则、命中行、预计章节数、异常提示
Tab 操作：确认切分、重新预览、规则保存
Tab 日志：split run 记录、PREVIEW_STALE 详情
```

按钮清单：
1. `按选中规则预览`
- 可见：始终可见
- 可用：存在启用规则且不在 loading
- loading：显示 spinner

2. `确认切分`
- 可见：有 preview 结果时
- 可用：preview 存在且未 stale
- 失败状态：提示“预览已过期，请重新预览”

3. `保存规则`
- 可见：始终
- 可用：仅有变更时
- 无变更：disabled

### 4.2 Screen B：Analyze（分析）

```
左栏：章节状态（未分析/已分析/失败）
中栏：原文 + 分析结果（摘要、人物、事件、场景）双层视图
右栏：
  洞察：识别置信度、疑似误判、缺失字段
  操作：重跑本章、应用编辑、标记通过
  日志：Analyze prompt 历史
```

按钮清单：
1. `重跑本章`
2. `保存分析编辑`
3. `标记通过`
4. `批量重跑失败章节`

### 4.3 Screen C：Mark（标记）

```
左栏：章节 + 标记完成度
中栏：原文段落 + 标记气泡（场景、策略、target_ratio）
右栏：
  洞察：规则命中路径（触发条件 -> 场景 -> 改写策略）
  操作：新增标记、删除标记、应用建议
  日志：自动标记与人工调整审计
```

按钮清单：
1. `应用自动标记`
2. `新增手动标记`
3. `保存标记`
4. `回滚到自动标记版本`

### 4.4 Screen D：Rewrite（改写）

```
左栏：章节 + 改写覆盖率 + 失败计数
中栏：
  A. 全章主画布：
     - 视图切换：原文 / 改写稿 / Diff（并排|行内）
     - Diff 样式：红删绿增，支持词级高亮
     - 改写稿头部显示：成功段数 / 失败段数 / 无可用改写空结果态
  B. 分段编辑区（中栏内，不在右栏）：
     - 选中场景卡后显示完整原文（不截断）
     - 显示改写结果全文与微调输入框
     - 段级按钮：接受 / 拒绝 / 保存微调 / 重写本段
右栏：
  洞察：策略、耗时、token、ANCHOR 风险、长度校验风险（REWRITE_LENGTH_OUT_OF_RANGE 的 target/min/max/actual）
  操作：仅章级/批量动作（执行本章、重跑本章、批量接受可用段）
  日志：每段调用历史 + 错误码 + 完整错误详情入口
```

按钮清单：
1. `接受（段级）`
2. `拒绝（段级）`
3. `微调并保存（段级）`（状态转为 `accepted_edited`）
4. `重写本段（段级）`
5. `执行当前章节（章级）`
6. `重跑当前章节（章级）`
7. `批量接受可用段（章级）`

### 4.5 Screen E：Assemble / Export（组装导出）

```
左栏：章节覆盖检查（已覆盖/回退/失败）
中栏：最终稿预览（可切换章节）+ 全书统计
右栏：
  洞察：质量闸门阈值对比、风险签名
  操作：普通导出 / 强制导出 / 导出格式选择
  日志：assemble run 历史、导出 manifest
```

按钮清单：
1. `执行组装`
2. `导出 TXT`
3. `导出 EPUB`
4. `强制导出（仅 blocked 时）`

## 5. 逐状态规范（UI 状态机）

### 5.1 Stage 级状态

1. `pending`
- 主按钮文案：`开始`
- 可操作：允许运行

2. `running`
- 主按钮文案：`运行中...`
- 可操作：允许暂停，不允许重复运行

3. `paused`
- 主按钮文案：`继续`
- 可操作：允许恢复

4. `completed`
- 主按钮文案：`重新执行`
- 可操作：允许导出/查看日志

5. `failed`
- 主按钮文案：`重试`
- 可操作：允许查看错误详情

6. `stale`
- 主按钮文案：`刷新该阶段`
- 可操作：阻止下游正式确认，提示先刷新

### 5.2 章节级状态（左栏）

1. `未处理`：灰点
2. `处理中`：蓝色动效
3. `完成`：绿色勾
4. `失败`：红色感叹号
5. `跳过`：黄色短杠
6. `需确认`：橙色问号

### 5.3 按钮级状态

每个关键按钮统一四态：
1. `default`
2. `hover`
3. `disabled`（无变更/前置条件不满足）
4. `loading`（spinner，禁止重复点击）

### 5.4 Rewrite 失败态（新增）

1. `REWRITE_LENGTH_OUT_OF_RANGE`
- 左栏：章节显示 `failed` 风险徽章。
- 中栏：改写稿头部显示“长度不达标”失败提示，并标记失败场景卡。
- 右栏洞察：显示 `target_ratio / target_chars_min / target_chars_max / actual_chars` 与建议动作。

2. `SEGMENT_FAILED_BUT_HAS_TEXT`
- 中栏分段编辑区允许查看完整原文与完整改写返回文本，支持人工微调后二次保存。
- 右栏不重复显示正文，只给状态和快捷动作。

## 6. 对比视图（Git 风格）

颜色语义：
1. 删除：`红底 + 删除线`
2. 新增：`绿底`
3. 未变：默认背景，可折叠大段未变内容

模式：
1. 并排模式：左原文右改写
2. 行内模式：同列显示 +/- 块

交互：
1. 点击差异块定位到段落锚点。
2. 锚点不一致时右栏显示 `ANCHOR_MISMATCH` 并禁用“接受”。

## 7. 右栏内容规划（最终版）

右栏不是“附属信息”，而是决策中心，固定承担三类职责：
1. `洞察`：解释“系统为什么这样判断”。
2. `操作`：提供“下一步做什么”的最短路径。
3. `日志`：提供“出错时如何追查”的完整链路。

各阶段最小字段：
1. Split：规则命中明细、边界摘要、preview_token 状态。
2. Analyze：summary 质量指标、角色/事件计数、缺失告警。
3. Mark：触发条件命中链路、策略冲突提示。
4. Rewrite：segment 状态分布、失败码分布、`REWRITE_LENGTH_OUT_OF_RANGE` 诊断字段（target/min/max/actual）、重试建议。
5. Assemble：覆盖率、failed ratio、risk_signature、导出阻断原因。

## 8. 与现有流程衔接

保持不变：
1. Split 仍是“预览 -> 确认”两步。
2. Rewrite 仍保留 `accepted_edited` 审计轨迹。
3. Assemble 仍受质量闸门控制。

新增约束：
1. Stage 级与批量级动作优先在右栏 `操作` Tab 完成。
2. Rewrite 的段落级审核与微调必须在中栏完成（完整原文 + 完整改写 + 微调输入）。
3. 右栏禁止承载正文编辑输入，避免中右重复信息与状态割裂。

## 9. 实施顺序（Local First + Docker 顺延）

执行顺序强约束：
1. 先完成三栏工作台开发与本地联调（前后端本地启动）。
2. 再完成核心链路回归测试（Import -> Split -> Analyze -> Mark -> Rewrite -> Assemble）。
3. 最后才进入 Docker 化（镜像、编排、容器冒烟）。

结论：Docker 相关任务保持在最后阶段，不前置、不并行穿插。

## 10. 基于最新反馈的专项修订（2026-03-21）

问题 -> 设计修订：
1. `原文` 与 `改写稿` 视觉上几乎一致。
- 修订：严格区分两者数据源；改写稿在“全失败/无有效段”时显示空结果态和失败统计，不回退为原文长文。

2. 改写失败细节只在右栏可见，中栏无法完整审阅。
- 修订：中栏新增分段编辑区，固定承载完整原文、完整改写、微调输入与段级动作。

3. 右栏出现“原文截断 + 改写全文”，信息分布割裂。
- 修订：右栏去正文化，只保留洞察、章级动作与日志入口。

4. 错误排查链路不完整。
- 修订：日志 Tab 增加“完整错误详情”入口，支持查看完整错误体（包含 provider 返回与校验上下文）。
