## Why

网络小说改写和扩写是一个高频需求，但手动逐章处理效率极低。需要一个自动化工具，能够导入小说后自动切分章节、识别场景、标记可改写区域，并利用大模型进行智能改写与扩写，大幅提升创作效率。Python 3.13 + FastAPI 后端提升开发迭代速度，React 19 前端提供流畅的操作体验。

## What Changes

- 以后端为主完成全栈落地：Python 3.13 + FastAPI 新后端 + 现有 React 19 前端完善与联调
- 支持导入 txt/epub 格式小说文件，自动解析内容
- 自动章节切分引擎，基于规则和 LLM 混合识别章节边界
- 章节切分支持“内置章标题规则 + 用户自定义正则”，支持规则测试预览
- 章节切分增强安全与一致性：regex 防 ReDoS 超时保护 + preview_token 防漂移确认
- LLM 驱动的章节深度分析：摘要生成 + 登场人物识别（情绪/状态追踪）+ 关键事件识别 + 可改写场景评估
- 可配置的场景识别系统（默认空白，场景类型与规则由用户手动添加），含改写潜力评估
- 智能改写标记：结合分析阶段的改写建议，自动标记可扩写/改写的段落
- LLM 驱动的内容改写与扩写引擎
- 改写后小说的完整导出（保持原始格式）
- 多模型支持：OpenAI 与 OpenAI 兼容提供商（如硅基流动），可灵活切换
- 并行 Worker 池：可配置并发数量，控制 LLM 调用吞吐
- 任务管理系统：进度追踪、暂停、重试
- 用户可自定义场景识别规则和改写策略
- 配置支持 JSON 导入导出，前端提供可视化配置编辑器
- 全局提示词配置：统一控制分析与改写风格（唯一提示词入口）
- 提供商管理增强：获取模型列表、模型名模糊搜索、按已选模型测试连接、同凭证 provider 保存时自动更新
- 前端交互统一为三栏工作台（左章节导航 / 中主内容 / 右洞察-操作-日志），并提供逐屏低保真线框基线
- 交付流程采用 Local First：先前后端本地联调验收，再进行 Docker 化部署
- 运行可靠性增强：Stage 配置快照、单飞锁+幂等触发、Assemble 质量闸门与回退告警
- 运行可追溯增强：保留 Stage 多次运行历史（run_seq），支持最新/历史 run 查询
- 改写审核增强：支持 accepted 后人工微调并保留审计轨迹
- 强制导出增强：写入风险签名（risk_signature）到质量报告与导出元数据

## Capabilities

### New Capabilities

- `novel-import`: 小说文件导入与解析（txt/epub），提取纯文本内容
- `chapter-splitting`: 章节自动切分引擎，支持内置规则、用户自定义 regex 和 LLM 辅助识别
- `chapter-summary`: 基于 LLM 的章节深度分析——摘要、登场人物（情绪/状态）、关键事件（类型/重要度）、改写潜力评估
- `scene-recognition`: 可配置的场景类型识别系统，支持自定义场景规则，含人物状态可视化和事件时间线
- `rewrite-marking`: 改写区域标记，根据场景规则识别可扩写/改写的段落
- `content-rewrite`: LLM 驱动的内容改写与扩写引擎
- `novel-export`: 改写后小说的完整导出，保持格式一致
- `llm-integration`: 多模型适配层，统一接口对接 OpenAI 与 OpenAI 兼容提供商，支持“全局提示词 + 内置阶段任务模板”与 provider 级参数管理
- `worker-pool`: 并行 Worker 管理，可配置并发数、队列调度、速率控制
- `task-management`: 任务生命周期管理，进度追踪、暂停、恢复、重试
- `config-management`: 用户配置管理，仅覆盖全局提示词、场景识别规则、改写规则，并支持 JSON 导入导出
- `frontend-app`: React 19 前端应用，Apple 风格 UI，任务操作、可视化配置编辑器、章节分析结果展示（人物卡片/事件时间线/改写建议图层）

## Impact

- **新增后端服务**：Python 3.13 Web 服务（FastAPI 框架），包含 REST API、WebSocket 进度推送、任务调度器
- **前端完善**：基于现有 React 19 SPA（Vite），补齐与后端联调、配置工作台和导出交互
- **外部依赖**：OpenAI API、OpenAI 兼容 API（如硅基流动）
- **存储**：SQLite 用于任务和配置持久化，文件系统用于小说文件存储
- **部署**：先本地联调与生产模式验收，最终再落地 Docker 单机部署（后端服务 + 前端静态资源）
